## 1. What This Repository Is

This is a **fork** of an open-source iOS app called **Open Relay** (originally "Open UI"),
created by GitHub user `Ichigo3766`. The original repo is at:

```
https://github.com/Ichigo3766/Open-Relay
```

Open Relay is a native iOS client for **Open WebUI**, which is a self-hosted AI chat
server (similar to ChatGPT but running on your own hardware). The app lets iPhone and
iPad users connect to their Open WebUI server and chat with AI models.

This fork renames the app to **ChatFort**, designed by **donphi**. The fork's GitHub repo is:

```
https://github.com/donphi/chatfort-ios-chat-relay
```

The fork is kept in sync with upstream. Brand customizations are managed by Python scripts
in the `tools/BrandOverride/` directory, not by manual file edits.

## 4. Build System: How project.pbxproj Works

The file `Open UI.xcodeproj/project.pbxproj` is the single most important configuration
file. It is a proprietary Apple format (old-style plist) that defines:

- **Targets:** `Open UI` (app) and `OpenUIWidgetsExtension` (widget)
- **Build settings:** Duplicated for Debug and Release configurations
- **File references:** Every source file, asset, and resource
- **Dependencies:** SPM package references

**Critical build settings for branding (all in pbxproj):**

| Setting | Current Value | Purpose |
|---------|---------------|---------|
| `INFOPLIST_KEY_CFBundleDisplayName` | `"Open Relay"` (app), `OpenUIWidget` (widget) | Name shown on home screen |
| `PRODUCT_BUNDLE_IDENTIFIER` | `com.openui.openui` (app), `com.openui.openui.OpenUIWidget` (widget) — overridden to `com.chatfort.chatfort` / `com.chatfort.chatfort.OpenUIWidget` by `brand_config.json` | Unique app identifier |
| `ASSETCATALOG_COMPILER_APPICON_NAME` | `AppIcon` | Which asset catalog icon set to use |
| `ASSETCATALOG_COMPILER_GLOBAL_ACCENT_COLOR_NAME` | `AccentColor` | Which asset catalog color to use as accent |
| `MARKETING_VERSION` | `2.4` | Version shown to users |
| `CURRENT_PROJECT_VERSION` | `1` | Build number |
| `DEVELOPMENT_TEAM` | `6ARX8PF3MT` | Apple Developer Team ID |
| `INFOPLIST_KEY_NS*UsageDescription` | Various "Open Relay needs..." strings | iOS permission prompts |

**IMPORTANT:** Every build setting appears TWICE in pbxproj — once for Debug, once for
Release. When making replacements, both occurrences must be changed. The override script
handles this automatically because it does global find/replace on the file.

**WARNING:** The pbxproj file is extremely sensitive to formatting. Never add or remove
lines. Only change the content of existing string values. A malformed pbxproj will prevent
Xcode from opening the project.

## 7. Theme Architecture

The app uses a custom theme system. Here is the complete data flow:

```
ColorTokens.swift          Defines raw hex colors for light and dark modes
       ↓                   (struct ColorTokens with ~40 color properties)
       ↓
AppTheme.swift             Combines ColorTokens with user's accent choice
       ↓                   (struct AppTheme, injected via @Environment(\.theme))
       ↓
AppearanceManager.swift    Manages user preferences (scheme, accent, pure black, tinted)
       ↓                   (@Observable class, persists to UserDefaults)
       ↓
Every SwiftUI View         Reads theme via: @Environment(\.theme) private var theme
                           Uses: theme.textPrimary, theme.background, theme.accentColor, etc.
```

### ColorTokens.swift

Location: `Open UI/Shared/Theme/ColorTokens.swift`

Defines a `ColorTokens` struct with ~40 color properties organized into categories:
Neutral, Brand, Accent, Status, Overlay, Scrim, Code, Channel Mentions, Reply Threads,
File Attachments. Has two static factories: `.light` and `.dark`, each with hardcoded
hex values via `Color(hex: 0xRRGGBB)`.

### AppTheme.swift

Location: `Open UI/Shared/Theme/AppTheme.swift`

Combines `ColorTokens` with the user's accent color choice. Provides semantic color
properties like `textPrimary`, `textSecondary`, `background`, `surfacePrimary`,
`accentColor`, etc. Injected into the SwiftUI environment via a custom `EnvironmentKey`.

### AppearanceManager.swift

Location: `Open UI/Core/Services/AppearanceManager.swift`

An `@Observable` class that persists user preferences:
- `colorSchemeMode`: system / light / dark
- `accentPreset`: one of 12 presets (monochrome, blue, indigo, purple, violet, pink, red, orange, amber, green, teal, cyan)
- `usePureBlack`: OLED black background in dark mode
- `useTintedBackgrounds`: tint surfaces with accent color
- `customAccentHex`: optional custom hex color

Each preset defines three hex colors: `color` (picker swatch), `lightColor` (light mode accent), `darkColor` (dark mode accent), `onAccentColor` (text on accent).

### DesignTokens.swift

Location: `Open UI/Shared/Theme/DesignTokens.swift`

Defines spacing (8pt grid), corner radii, icon sizes, touch targets, animation durations,
and opacity levels as static constants in enums.

### Typography.swift

Location: `Open UI/Shared/Theme/Typography.swift`

Defines font scale using SF system fonts with Dynamic Type support. No custom fonts.

## 8. Asset Catalog Internals and Icon Composer

### How .xcassets Works

An `.xcassets` folder is an Xcode Asset Catalog. Inside it, each subfolder ending in
`.appiconset`, `.imageset`, or `.colorset` is a "set" containing:
- A `Contents.json` file that describes the assets
- The actual asset files (PNGs, etc.)

### Icon Composer (.icon) Replaces AppIcon.appiconset

The app icon now uses an **Icon Composer `.icon` bundle** instead of the traditional
`AppIcon.appiconset`. When a `.icon` file named `AppIcon.icon` is present in the
Xcode project, it takes precedence over any `AppIcon.appiconset` in the asset catalog.

The `.icon` bundle is a folder containing:
- `icon.json` — layer definitions, colors, Liquid Glass effects, appearance variants
- `Assets/` — SVG or PNG layer artwork files

Xcode generates all icon variants at build time (all sizes, all platforms, all
appearances including light/dark/mono). It also auto-generates backward-compatible
flat icons for iOS 18 and earlier.

The upstream `AppIcon.appiconset` stays in the repo untouched. When the override
script copies `AppIcon.icon` into the project, it takes precedence. When the restore
script removes it, the upstream `appiconset` resumes control.

### AppIconImage.imageset/Contents.json

This is a regular image asset used for `Image("AppIconImage")` in SwiftUI views
(About screen, Onboarding, Login, Widgets). SwiftUI cannot read `.icon` files
directly, so this imageset still uses a PNG. The override script copies
`AppIcon-preview.png` (exported from Icon Composer) into the 1x slot.

## 9. Localization Internals

### How Localizable.xcstrings Works

`Open UI/Localizable.xcstrings` is a JSON file (Apple's String Catalog format). Structure:

```json
{
  "sourceLanguage": "en",
  "strings": {
    "Some English Key": {
      "localizations": {
        "ar": { "stringUnit": { "state": "translated", "value": "Arabic translation" } },
        "de": { "stringUnit": { "state": "translated", "value": "German translation" } },
        ...
      }
    }
  }
}
```

The English key IS the source string. When SwiftUI encounters `Text("Some English Key")`,
it looks up that exact key in the catalog and returns the translation for the current locale.

**Why the scripts skip this file:**
- It contains ~168 lines with "Open Relay" or "Open UI" spread across ~30 languages
- Each upstream update frequently modifies this file (new strings, updated translations)
- Changing it creates massive merge conflicts
- The English strings are already changed by modifying the Swift source files (the key
  lookup still works because SwiftUI falls back to the literal string if no key matches)

## 10. Override System Architecture

### brand_config.json

Location: `tools/BrandOverride/brand_config.json`

This JSON file is the single source of truth for all brand values. It contains:

- `brand`: Object with app name, author, GitHub URLs, widget text, credits
- `string_replacements`: Array of objects, each with a `file` path and `replacements` array
  of `{find, replace}` pairs
- `icon_composer`: Object with `source` (path to `.icon` bundle) and `targets` (array of
  destination paths in the project)
- `icon_files`: Array of `{source, target}` pairs for the preview PNG (in-app display)
- `files_to_backup`: Array of relative file paths that the backup script should snapshot

### Script Execution Flow

**backup.py:**
1. Load `brand_config.json`
2. Read `MARKETING_VERSION` and `CURRENT_PROJECT_VERSION` from `project.pbxproj`
3. Get git short hash via `git rev-parse --short HEAD`
4. Create `backups/v{ver}_build{b}_{date}_{hash}/pristine/` and `override/`
5. Copy each file from `files_to_backup` into both subdirectories
6. Print summary

**restore.py:**
1. Find latest backup directory (or use `--version` argument)
2. For each file in `files_to_backup`:
   - Compare `pristine/` copy with current repo file
   - If different, copy `pristine/` version back to repo
   - Show diff of what changed
3. Print summary

**override.py:**
1. Load `brand_config.json`
2. For each entry in `string_replacements`:
   - Read the target file
   - Apply all find/replace pairs
   - Collect change details (line number, before/after)
3. Copy the Icon Composer `.icon` bundle to each target in `icon_composer.targets`
4. If `AppIcon-preview.png` is missing, attempt auto-export via `ictool` (Xcode 26+)
5. For each entry in `icon_files`:
   - Check if source exists in `tools/BrandOverride/Assets/`
   - If `--apply`, copy source to target
6. Print colored diff of all changes
7. If `--apply`, write modified files to disk

### Colored Diff Output

The scripts use ANSI escape codes for terminal colors:
- `\033[91m` = Red (removed text)
- `\033[92m` = Green (added text)
- `\033[93m` = Yellow (line numbers, warnings)
- `\033[96m` = Cyan (file headers, separators)
- `\033[1m` = Bold
- `\033[2m` = Dim
- `\033[0m` = Reset

No external Python packages are required. Everything uses the standard library.

## 11. Merge/Sync Workflow

### Pulling Upstream Updates

```bash
# Step 1: Revert all brand changes
cd "tools/BrandOverride"
./scripts/restore.sh

# Step 2: Pull upstream
cd ../..
git pull upstream main

# Step 3: If there are merge conflicts, resolve them
# (conflicts should be rare since restore.sh reverted our changes)

# Step 4: Create a new backup of the updated files
cd "tools/BrandOverride"
./scripts/backup.sh

# Step 5: Preview what the override will do
./scripts/override.sh --dry-run

# Step 6: Check the dry-run output carefully:
#   - Are all expected changes still present?
#   - Did any line numbers shift? (The script finds by string content, not line number)
#   - Are there any "File not found" warnings? (File may have been renamed upstream)
#   - Are there any find strings that no longer match? (Upstream may have reworded something)

# Step 7: Apply
./scripts/override.sh --apply

# Step 8: Build and test in Xcode
```

### If a Find String No Longer Matches

If upstream changed a branded string (e.g., rewrote a permission prompt), the override
script will silently skip that replacement (no match = no change). The dry-run output
will show fewer changes than expected. To fix:

1. Check `CANNOT_OVERRIDE.md` for the expected string
2. Search the upstream file for the new wording
3. Update the `find` value in `brand_config.json`
4. Re-run `./scripts/override.sh --dry-run` to verify

### If a File Was Renamed or Moved

The script will print a "File not found" warning. To fix:

1. Find the new file path in the repo
2. Update the `file` value in `brand_config.json`
3. Update the path in `files_to_backup` too
4. Update `CANNOT_OVERRIDE.md` with the new path

## 13. Common Pitfalls

### pbxproj Corruption

The `project.pbxproj` file uses a specific format. If you:
- Add or remove lines → Xcode cannot parse the file
- Break a string's quoting → Xcode cannot parse the file
- Change a setting's key name → Build fails

The override script only does exact string replacement within existing lines, which is safe.

### Asset Catalog and Icon Composer

The override script does NOT modify any `Contents.json` files. The app icon is handled
entirely by the Icon Composer `.icon` bundle, which Xcode processes at build time. The
only PNG the script touches is the in-app preview image (`AppIconImage.imageset`).

If you see icon issues after an override:
- Verify `AppIcon.icon` was copied into `Open UI/` and `OpenUIWidgets/`
- Verify the target's "App Icon Set Name" build setting is `AppIcon`
- Clean the build folder (Shift+Cmd+K) to clear cached icon assets

### Code Signing

## 14. Alternative Tool: patchy

[patchy](https://github.com/nik-rev/patchy) is a Rust-based tool for declaratively
managing personal forks. Install via `brew install patchy-bin`.

It works by:
1. Defining a TOML config with your upstream repo and custom patches
2. Running `patchy run` to sync upstream and auto-apply patches
3. Handling merge conflicts automatically where possible

This could replace the Python scripts if the project grows more complex. The tradeoff
is adding a Rust dependency and learning a different workflow. The Python scripts are
self-contained and require only Python 3 (pre-installed on macOS).

## 15. Quick Reference: Running the Scripts

All commands assume you are in the repo root directory.

```bash
# Create backup
cd "tools/BrandOverride" && ./scripts/backup.sh

# Restore to upstream
cd "tools/BrandOverride" && ./scripts/restore.sh

# Preview brand changes
cd "tools/BrandOverride" && ./scripts/override.sh --dry-run

# Apply brand changes
cd "tools/BrandOverride" && ./scripts/override.sh --apply

# Restore a specific backup version
cd "tools/BrandOverride" && ./scripts/restore.sh --version v2.4_build1_2026-04-01_f49448d

# Backup with a custom label
cd "tools/BrandOverride" && ./scripts/backup.sh --tag before-v2.5-merge
```

## 16. Checklist for LLMs Making Changes

Before modifying any file in this repo:

1. **Is the file in `tools/BrandOverride/`?** If yes, edit freely.
2. **Is the file an upstream file?** If yes, do NOT edit it directly. Instead:
   a. Add the change to `brand_config.json` in the `string_replacements` section
   b. Add the file to `files_to_backup` if not already there
   c. Update `CANNOT_OVERRIDE.md` with the new entry
   d. Run `./scripts/override.sh --dry-run` to verify
   e. Run `./scripts/override.sh --apply` to write
3. **Is it a new brand value?** Add it to `brand_config.json` under `brand`.
4. **Is it a theme/color change?** These are in `ColorTokens.swift` and
   `AppearanceManager.swift`. They are NOT in `brand_config.json` because they are
   not simple string replacements. Document the change in `CANNOT_OVERRIDE.md`.
5. **After any change**, run `./scripts/override.sh --dry-run` to verify the full
   override still works correctly.
